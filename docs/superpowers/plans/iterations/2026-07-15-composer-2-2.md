## Objective and acceptance

Implement the approved Composer 2.2 plan and prototype as a sequence of independently observable commits while preserving NeatChat's existing provider, session, attachment, submission, persistence, and security semantics.

The complete implementation is accepted only when:

- Compact, Expanded, and Scrolling are content-driven states of one Composer Shell rather than separate visual surfaces.
- Tools, Prompt Library, Model Chip, parameter Rails, Send/Stop, menus, focus, motion, theme, Container Query, Safe Area, and fold fallback match the approved behavior.

- Relevant Jest, ESLint, TypeScript, `git diff --check`, production build, and real Browser viewport/theme/state matrices pass.
- Every implementation slice updates this handoff and lands in a local commit; no work is pushed.

## Slice 4 — adaptive placement, semantic visual system, and posture handling

Status: implementation, Root verification, Browser QA, and independent re-review complete; this handoff accompanies the local Slice 4 commit based on `5c4c015d`.

### Observable result

- Tools and Model-family panels now use one `getComposerPopoverPlacement` contract. It accepts the real trigger, Composer rect, measured panel height, `visualViewport`, Safe Area, and viewport segments; it chooses the preferred side only when that side fits, otherwise flips or uses the larger remaining side, clamps width/height, and exports one CSS-variable contract.
- Prompt Library resize, model-layer transitions, zoom, orientation, visual viewport scroll/resize, and shell/panel size changes all schedule the same placement calculation through `requestAnimationFrame` plus `ResizeObserver`. Tools coordinates are converted into the input panel containing block; Model coordinates remain viewport-based.
- Book, Tabletop, Split, fold-outer, and fold-inner QA postures now create real segment probes. The same probe path is used by the platform `viewport-segment-*` media queries, so the Composer and every open panel stay inside the selected usable segment instead of crossing the hinge.
- The shell reads visual viewport top/height/bottom inset and selected segment top/height/bottom inset into CSS variables. Existing-conversation Composer follows the visible bottom edge; empty Composer keeps its centered desktop intent and moves to a safe bottom position for compact or vertical-segment layouts.
- Compact, medium, and wide Composer control widths are now expressed through `@container chat-composer` rules rather than a JavaScript device breakpoint. The existing runtime measurement only derives placement and viewport geometry; it does not fork business behavior or DOM.
- Light and Dark expose one complete semantic Composer token set in `globals.scss`. Shell, Add, Chip, Send, Tools, Prompt Library, Model layers, selected/disabled states, and Reasoning Rail consume those tokens; the corresponding component-local complete Dark/System token copies are removed.
- Aura is one local pseudo-element around the Composer shell. It uses three semantic radial gradients, never receives pointer input, reacts to focus, ready/send, uploading, and Stop, and only animates opacity/transform. Dark tokens reduce its intensity; Reduced Transparency removes it.
- Reduced Motion removes Aura/Stop/Rail animation and caps Composer transitions at `80ms`. Reduced Transparency removes backdrop blur and uses opaque semantic panel surfaces. High Contrast strengthens boundaries; Forced Colors removes Aura/gradient/shadow paint and restores Canvas/CanvasText/Highlight focus semantics.
- The development-only `codex_qa=composer` fixture now drives the real input, attachment strip, accepted/blocked dropzone, Tools/Prompt/Model layers, focus, uploading, and Disabled/Send/Stop states for 20 named cases. Non-empty states render deterministic in-memory messages and do not write unfinished drafts or mutate persisted sessions.
